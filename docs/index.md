# Wamuale Development Client Handoff Guide

This guide explains how to operate the Wamuale Development housing project CRM and how staff should move a buyer from first inquiry through application review, lot reservation, contract management, payments, collections, and post-sale follow-up.

!!! note "Final branding pending"
    The guide currently uses **Wamuale Development** and the proposed documentation address `docs.ameladevelopment.com`. The title, logo, colors, and URL should be updated after the client confirms the final brand guide and domain.

## 1. What the System Manages

The CRM provides one operational home for:

- Available, reserved, and sold lots
- Buyer inquiries and sales follow-ups
- Applications and approval decisions
- Reservations and deposit readiness
- Customers and contracts
- Payments, receipts, balances, and collections
- Post-sale tasks and accountability
- Daily priorities, reports, emails, and audit history

The core operating principle is simple: **every buyer should have a clear current stage, a clear next action, and an accountable staff member.** Staff should update the CRM as work happens so the Dashboard, Daily Brief, reports, and Smart Insights reflect the real state of the business.

## 2. Roles and Responsibilities

### Super Admin

Intended for the business owner or person responsible for the complete system.

Typical responsibilities include:

- Managing users and roles
- Configuring company, payment, lot, fee, installment, reservation, email, and AI settings
- Reviewing sensitive financial and audit information
- Correcting configuration issues
- Approving major process changes

### Admin

Intended for trusted operational managers.

Typical responsibilities include:

- Managing leads, applications, customers, contracts, and payments
- Reviewing daily priorities and reports
- Managing the Email Center
- Monitoring collections and post-sale work
- Correcting operational data

### Staff

Intended for sales, customer service, collections, or operations personnel.

Typical responsibilities include:

- Updating lead stages and activities
- Completing assigned follow-ups
- Scheduling and recording site visits
- Reviewing applications within assigned responsibilities
- Recording customer interactions and documents
- Updating post-sale tasks

### Read Only

Intended for management or reviewers who need visibility without the ability to change records.

!!! warning "Use the lowest necessary access"
    Do not give Super Admin access simply because it is convenient. Access should match each person's actual responsibilities.

## 3. First Login and Initial Setup

Before staff begin entering live information, a Super Admin should confirm:

- Company profile details are correct.
- Active housing projects and lots have been entered.
- Lot prices, sizes, zoning, and statuses are accurate.
- Payment methods and fee types are configured.
- Installment plans match approved company terms.
- Reservation and deposit defaults are correct.
- Staff accounts have the correct roles.
- Notification and email settings have been reviewed.

### Controlled Test Before Launch

Complete one end-to-end test using a clearly identified test buyer:

1. Submit a public application or create a lead.
2. Add a follow-up activity.
3. Schedule a site visit.
4. Reserve a test lot.
5. Review and approve the test application.
6. Create the customer and contract records.
7. Record a test payment and verify the balance.
8. Complete or cancel all test records before launch.

This verifies that the full workflow is understood before real customer data is entered.

## 4. Recommended Daily Workflow

### Step 1: Start with the Dashboard

Review **Today's Attention** before opening individual records. Look for:

- Overdue or urgent follow-ups
- Reservations nearing expiry
- Deposits awaiting proof or confirmation
- Site visits scheduled for today
- Applications waiting for review
- Collections alerts
- Blocked or overdue post-sale tasks

### Step 2: Generate or Review the Morning Brief

Use the Daily Brief to summarize the most important operational issues. Convert meaningful items into actions and update the underlying records as work is completed.

The brief is guidance. Important facts should always be confirmed in the related lead, customer, payment, reservation, or contract record.

### Step 3: Work the Lead Queue

For every active lead:

1. Confirm the current pipeline stage.
2. Review the last contact and next action.
3. Complete or reschedule overdue follow-ups.
4. Log calls, WhatsApp messages, emails, notes, and site visits.
5. Update the stage when the buyer moves forward or becomes inactive.

### Step 4: Review Applications

Prioritize applications that are pending, incomplete, linked to an unavailable lot, or flagged as possible duplicates.

Before approval:

- Verify applicant information.
- Confirm the selected lot remains available.
- Review linked lead and reservation information.
- Resolve missing information.
- Use the AI review as supporting guidance, not as the final decision.

### Step 5: Check Reservations and Deposits

Review reservations that are expiring or waiting for deposits. Confirm submitted proof before marking a deposit as confirmed. Release lots only through the approved company process.

### Step 6: Record Payments and Collections Activity

Record payments promptly and accurately. Verify the customer, contract, payment type, amount, date, method, receipt number, and supporting document before saving.

Follow up on overdue accounts and update payment requests when their status changes.

### Step 7: Complete Post-Sale Work

Review open, overdue, blocked, and urgent post-sale tasks. Keep agreement, document, payment setup, and collections handoff statuses current.

### Step 8: Close the Day Cleanly

Before finishing:

- Complete or reschedule due follow-ups.
- Add notes to records that changed.
- Confirm payments entered that day.
- Resolve avoidable missing receipt numbers or documents.
- Update blocked post-sale tasks with a reason.
- Submit system issues through **Send Feedback**.

## 5. Sales Workflow

### Standard Buyer Journey

```text
New Lead → Contacted → Interested → Family Decision → Payment Plan Review
→ Site Visit Scheduled → Deposit Pending → Deposit Paid
→ Application Started → Contract Started → Closed Won
```

A buyer may skip stages when appropriate, but the selected stage should always describe their real position.

### Lead Intake

Leads may come from a public application or be entered by staff. Check possible duplicate warnings before creating or working multiple records for the same person.

Capture:

- Full name
- Phone and email
- Preferred project or lot
- Budget or payment-plan interest
- Source of inquiry
- Assigned staff member
- Next follow-up date

### Activities and Follow-Ups

Log meaningful interactions as they happen, including notes, calls, WhatsApp messages, emails, status changes, site visits, and follow-ups.

Every active lead should have a next action unless the buyer is closed, lost, or intentionally paused.

### Site Visits

When scheduling a visit:

1. Confirm the buyer, date, time, and location.
2. Add any preparation notes.
3. After the visit, mark it completed, rescheduled, cancelled, or no-show.
4. Add the outcome and next step to the lead activity history.

### Lot Reservations

A reservation is a temporary operational hold, not a completed sale.

Before creating one:

- Confirm the lot is available.
- Confirm the buyer's interest.
- Set the expiry date and expected deposit amount.
- Explain the company's reservation terms.

Update reservation and deposit statuses as evidence is received and verified.

### Application Review and Approval

Approval should happen only after staff confirm the applicant's information and lot choice. Do not approve solely because the AI review appears positive.

## 6. Customer, Contract, and Collections Workflow

### Customer Record

The customer profile becomes the main operational record after approval. It combines identity, lot, contract, payments, requests, documents, post-sale work, and account guidance.

Use the status strip and operational summary for orientation, then open the relevant tab for detailed work.

### Contract Management

Before activating a contract, verify:

- Correct customer
- Correct lot
- Purchase price
- Initial deposit
- Installment amount and term
- Start date and due-date expectations
- Additional fees or conditions

Voiding or cancelling a contract is a sensitive action. Record a clear reason and follow the company's authorization process.

### Recording Payments

For every payment:

1. Confirm the customer and active contract.
2. Select the correct transaction type.
3. Enter the exact amount and date received.
4. Select the collection method.
5. Enter the manual receipt number where required.
6. Upload the transfer proof or receipt image where applicable.
7. Verify the updated account balance.

Never record a bank transfer as confirmed solely because a customer says it was sent. Verify the proof and the company's bank records.

### Payment Requests and Collections

Payment requests can be prepared, sent, marked paid, or cancelled. Keep the status current.

Collections work should prioritize:

- Overdue accounts
- Accounts due soon
- Missing payment proof
- Missing receipt numbers
- Disputed or unclear payments
- Customers with no active contract or inconsistent account data

Record the outcome of every collections contact in the appropriate customer, payment request, or activity record.

## 7. Feature Guide

### Dashboard

A daily operational overview showing items requiring attention, sales movement, reservations, deposits, collections, post-sale work, and supporting totals.

### Daily Brief

Creates a summarized morning view from current system data. It helps staff prioritize but does not replace review of the underlying records.

### Email Center

Allows authorized users to review notification history and email-related activity. Failed messages should be investigated rather than repeatedly resent without checking the cause.

### Leads

The sales workbench for inquiries, stages, activities, follow-ups, site visits, reservations, and buyer context.

### Lots

The inventory view for land parcels:

- **Available:** open for selection
- **Reserved:** temporarily held
- **Sold:** attached to a completed sale or active ownership process

### Applications

The review workspace for submitted applications. It supports applicant review, lot selection, missing-information checks, duplicate indicators, AI guidance, approval, and decline actions.

### Customers

The central post-approval record containing customer, contract, financial, document, request, and post-sale information.

### Contracts

The legal and financial agreement record for a lot sale. Contract changes should follow company authorization rules.

### Payments

Records confirmed financial transactions and supporting receipt or transfer information.

### Collections

Prioritizes accounts needing payment follow-up and shows due-date and balance context.

### Reports

Provides management views based on recorded system data. Report accuracy depends on staff keeping records current.

### Audit Trail

Shows accountable history for supported changes. Use it when reviewing who changed a record and when.

### Settings

Controls company information, payment methods, installment plans, lot sizes, fee types, reservation workflow, notifications, users, roles, and AI configuration.

### Send Feedback

Use this for bugs, questions, data issues, and feature requests. Include what you were doing, what happened, what you expected, and the affected record. Never include passwords or API keys.

## 8. Data Quality and Accountability

- Search before creating a new lead or customer.
- Review possible duplicate warnings.
- Use one primary customer record per buyer.
- Do not mark deposits or payments confirmed without evidence.
- Enter receipt numbers exactly as issued.
- Keep lot statuses synchronized with reservations and contracts.
- Record reasons for cancellations, releases, declines, and voids.
- Add notes that another staff member can understand later.
- Never place passwords, banking credentials, or private API keys in notes or feedback.

A possible duplicate warning is a prompt for staff review, not proof that two records are the same. Compare email, phone, name, and lot interest before deciding how to proceed.

Use the Audit Trail to investigate supported record changes. When correcting an error, add a clear note where the workflow permits.

## 10. Support and Feedback

When reporting an issue, include:

- The page or feature being used
- The affected buyer, lot, contract, or payment reference
- What action was attempted
- What happened
- What was expected
- Whether the issue blocks work
- A screenshot when it helps explain the issue

Priority guidance:

- **Low:** Cosmetic issue or suggestion with no workflow impact
- **Normal:** Standard question, data issue, or non-blocking bug
- **High:** Important workflow is impaired and needs prompt attention
- **Urgent:** Production is unavailable, financial data may be at risk, or critical work cannot continue

Before repeating a financial action, confirm whether the original action already saved.

## 11. Hosting This Guide on GitHub Pages

This guide uses MkDocs Material and can be hosted free through GitHub Pages.

### Recommended Address

After the final domain is confirmed, use a documentation subdomain such as:

```text
docs.ameladevelopment.com
```

A neutral alternative is:

```text
help.ameladevelopment.com
```

### GitHub Pages Setup

1. Merge the documentation pull request into `main`.
2. In the repository, open **Settings → Pages**.
3. Set the source to **GitHub Actions**.
4. The included workflow will build and publish the guide when documentation changes are pushed to `main`.
5. Open the generated Pages URL and verify the guide.

### Custom Subdomain Setup

After the domain is final:

1. Add the custom domain in **Settings → Pages**.
2. At the DNS provider, create a `CNAME` record for `docs` pointing to:

```text
edmundciego.github.io
```

3. Wait for GitHub's DNS check to pass.
4. Enable **Enforce HTTPS**.
5. Update `site_url` in `mkdocs.yml` if a different subdomain is selected.

!!! warning "Public documentation only"
    GitHub Pages is a public static website for this public repository. The guide must never contain customer records, passwords, API keys, internal credentials, or confidential financial documents.

Documentation is stored as Markdown and can be updated through pull requests, providing a clear review process and version history.
