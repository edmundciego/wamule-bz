# Phase 4: Smart Assistance + Reporting Polish Plan

This is a planning document only. It does not implement schema, frontend, Supabase, Edge Function, permission, route, payment, contract, collections, application approval, customer creation, reservation, post-sales, or workflow changes.

## Pending QA Item

Authenticated protected-route mobile/browser QA still needs to be completed for Dashboard, Lots, Applications, Customers, Customer Detail, Payments, Collections, Reports, Emails, Daily Briefs, Settings, modals, upload/proof areas, and dense tables once valid admin credentials are provided.

## 1. Current AI / Reporting Summary

Wamule already has an advisory AI and reporting foundation:

- Application AI Review:
  - Stored in `application_ai_reviews`.
  - Generated by `generate-application-review`.
  - Uses deterministic checks first and Gemini only when AI settings and Edge Function secrets are enabled.
  - Summarizes application completeness, missing fields, lot conflicts, risk flags, and recommended admin actions.
  - Admin-only generation; internal users can read.

- Daily Brief:
  - Stored in `ai_daily_briefs`.
  - Generated by `generate-daily-brief`.
  - Covers applications, lots, payments, contracts, collections, alerts, and recommended actions.
  - Creates or updates `brief_action_items` as carryover operational items.
  - Admin-only generation; internal users can read.
  - Current brief data does not yet include Phase 1-3 sales, reservation, deposit-readiness, or post-sales tables.

- Customer Summary / Collections Assistant:
  - Stored in `customer_ai_summaries`.
  - Generated by `generate-customer-summary`.
  - Summarizes account status, balance, payments, collections flags, missing items, recommended actions, and a draft follow-up message.
  - Uses deterministic fallback and optional Gemini when `collections_assistant_enabled` is active.
  - Staff/Admin/Super Admin can generate; internal users can read.

- Dashboard widgets:
  - Existing dashboard includes lot inventory, applications, revenue, balances, sales pipeline, follow-ups, site visits, reservations/deposit readiness, and post-sales readiness counts.
  - These widgets are live counts and rule-based indicators, not stored summaries.

- Reports page:
  - Current tabs cover Payments, Balances, Applications, Lots, and Missing Items.
  - CSV export already exists for existing report sections.
  - Reports do not yet include leads, follow-ups, site visits, reservations, deposit readiness, post-sales tasks, or staff workload.

- AI settings/provider configuration:
  - `ai_settings` controls provider, model, global AI enabled state, Daily Brief, Application Summary, and Collections Assistant toggles.
  - Provider is currently constrained to Gemini.
  - Secrets are configured in Supabase Edge Functions, not browser code.
  - Settings page includes provider health check and toggle controls.

## 2. Phase 4 Product Goal

Phase 4 should help staff see what needs attention across sales, reservations, deposits, applications, post-sales readiness, and collections without replacing staff judgment.

The goal is to:

- Summarize buyer/customer status in calm operational language.
- Surface next actions and blockers.
- Improve reporting for sales and post-sales accountability.
- Reduce manual review work by grouping the most important records.
- Keep staff in control of decisions, status changes, messages, approvals, payments, and contracts.

AI should support staff decisions, not make decisions. Deterministic rules should be used wherever the system already has enough structured data.

## 3. Smart Assistance Opportunities

### Lead Detail

- Summary: contact details, pipeline stage, buyer journey, preferred lot, next action, overdue follow-ups, site visits, linked application/customer, reservation state.
- Recommended actions: assign staff, schedule follow-up, schedule site visit, review deposit readiness, link application/customer, mark inactive if stale.
- Data needed: `leads`, `lead_activities`, `follow_up_tasks`, `site_visits`, `lot_reservations`, linked `applications`, linked `customers`, `parcels`.
- Rules enough: MVP can be rule-based.
- AI needed: later, for timeline synthesis and buyer readiness narrative from many notes.
- Phase: MVP rules; AI later.

### Sales Pipeline

- Summary: leads by stage, stale leads, overdue next actions, unassigned leads, source performance.
- Recommended actions: focus on overdue follow-ups, review family decision leads, confirm next action for unassigned/new leads.
- Data needed: `leads`, `follow_up_tasks`, `site_visits`, `lead_activities`.
- Rules enough: yes for counts, stale states, and overdue items.
- AI needed: not necessary for MVP.
- Phase: MVP.

### Follow-up Center

- Summary: due today, overdue, unassigned, high-priority, linked lead/application/customer context.
- Recommended actions: complete/cancel stale tasks, prioritize urgent overdue tasks, assign owner.
- Data needed: `follow_up_tasks`, `leads`, `applications`, `customers`, `admin_profiles`.
- Rules enough: yes.
- AI needed: later only for suggested call/email talking points.
- Phase: MVP or Phase 4C if a dedicated center is deferred.

### Reservation / Deposit Readiness

- Summary: active reservations, expiring soon, expired/manual review, deposit pending, proof submitted, confirmed, waived, overdue.
- Recommended actions: follow up before expiry, request deposit proof, review confirmed deposits for next step, release inactive holds.
- Data needed: `lot_reservations`, `reservation_activities`, `leads`, `applications`, `customers`, `parcels`, optional linked `transactions`.
- Rules enough: yes for MVP.
- AI needed: later for readiness narrative when notes/history are long.
- Phase: MVP rules.

### Application Review

- Summary: current AI review plus linked lead/reservation/post-sales status.
- Recommended actions: request missing info, resolve lot conflict, create/link lead, create reservation, proceed to approval only through existing manual flow.
- Data needed: `applications`, `application_ai_reviews`, `leads`, `lot_reservations`, `parcels`.
- Rules enough: existing deterministic checks cover most needs.
- AI needed: already optional through current Application AI Review.
- Phase: extend carefully in Phase 4B/4D.

### Customer Detail

- Summary: account status, contract/payment standing, documents, linked lead/reservation, post-sales checklist, collections readiness.
- Recommended actions: update post-sales checklist, follow up on payment, upload signed contract, confirm handoff, resolve missing proof/receipt.
- Data needed: `customers`, `applications`, `contracts`, `transactions`, `payment_documents`, `payment_requests`, `leads`, `lot_reservations`, `post_sales_checklists`, `post_sales_tasks`.
- Rules enough: many flags are rule-based.
- AI needed: useful later for a combined customer timeline summary.
- Phase: MVP rules; hybrid later.

### Post-Sales Checklist

- Recommended actions: request missing documents, review agreement, follow up for signature, hand off to collections, resolve blocked checklist.
- Data needed: `post_sales_checklists`, `post_sales_tasks`, `post_sales_activities`, linked customer/application/contract/lead/reservation.
- Rules enough: yes for MVP.
- AI needed: not necessary until activity timelines become long.
- Phase: MVP.

### Collections

- Summary: overdue accounts, due soon, missing receipts/proofs, open payment requests, missing signed contracts, handoff-ready customers.
- Recommended actions: contact overdue customer, request proof, enter receipt, generate payment request, review handoff-ready accounts.
- Data needed: `contracts`, `transactions`, `payment_documents`, `payment_requests`, `customer_balance_view`, `post_sales_checklists`.
- Rules enough: yes for operational alerts.
- AI needed: already partly covered by Customer Summary; later for account-level context.
- Phase: Phase 4B/4D.

### Daily Brief

- Summary: current operational brief plus sales activity, follow-ups, site visits, reservation/deposit readiness, post-sales blockers.
- Recommended actions: ranked staff priorities for the day.

- Rules enough: enough for MVP expansion.
- AI needed: optional after deterministic sections are added.
- Phase: Phase 4B.

### Dashboard

- Summary: highlight highest-priority next actions and stuck workflows.
- Recommended actions: click-through targets to Leads, Daily Briefs, Reports, Customer Detail.
- Data needed: existing dashboard query data plus report/brief outputs.

## 6. Smart Summary Design

### Lead Smart Summary

- Purpose: give staff a quick buyer-readiness and next-action view.
- Inputs: lead profile, pipeline stage, next action, follow-up tasks, site visits, activity timeline, linked application/customer/reservation.
- Output: short summary, readiness badges, risk flags, recommended next actions.
- Example actions: "Schedule follow-up", "Confirm preferred lot", "Review deposit readiness", "Assign a team member".
- Logic: rules for MVP; AI later for long timeline notes.

### Reservation Readiness Summary

- Purpose: clarify whether a reservation is stable, expiring, deposit-ready, or needs release.
- Inputs: reservation status, deposit status, due/expiry dates, lead/customer/application links, activities.
- Output: readiness state, deposit state, expiry warning, next action.
- Example actions: "Follow up before expiry", "Review submitted proof", "Confirm next application/contract step".
- Logic: rules for MVP; AI rarely needed.

### Customer Smart Summary

- Purpose: extend existing customer summary to include sales and post-sales context.
- Inputs: current customer summary inputs plus linked lead, reservation, post-sales checklist/tasks.
- Output: account summary, readiness summary, collections flags, post-sales blockers, recommended actions.
- Example actions: "Upload signed contract", "Request missing documents", "Review collections handoff".
- Logic: hybrid later; keep existing deterministic fallback.

### Post-Sales Checklist Summary

- Purpose: show readiness for agreement, documents, payment setup, and collections handoff.
- Inputs: checklist statuses, tasks, activities, linked customer/application/contract/lead/reservation.
- Output: readiness review, blocker list, overdue task list, next action.
- Example actions: "Request missing documents", "Review agreement before sending", "Hand off to collections".
- Logic: rules for MVP; AI later for activity-heavy accounts.

### Collections Summary

- Purpose: prioritize payment/collections attention without mutating payment records.
- Inputs: contracts, transactions, payment documents, payment requests, balances, post-sales handoff status.
- Output: due/overdue summary, missing proof/receipt flags, customer contact recommendation.
- Example actions: "Contact overdue customer", "Confirm transfer proof", "Review open payment request".
- Logic: hybrid already exists in Customer Summary; expand carefully later.

### Daily Operations Brief

- Purpose: one operational view of the day across sales, applications, reservations, payments, collections, and post-sales.
- Inputs: all existing Daily Brief inputs plus Phase 1-3 tables.
- Output: section summaries, alerts, recommended priorities, carryover action items.
- Example actions: "Review overdue follow-ups", "Confirm deposit proof", "Resolve post-sales blocker".
- Logic: deterministic first, optional Gemini summarization after fallback is complete.

## 7. AI Safety and Control

Phase 4 must keep these guardrails:

- AI must not approve applications.
- AI must not confirm deposits.
- AI must not modify payments.
- AI must not modify contracts.
- AI must not mark documents approved.
- AI must not change collections status.
- AI must not send WhatsApp/email messages automatically.
- AI must not expose private data publicly.
- AI output should be reviewable and clearly labeled.
- Staff remain responsible for decisions.
- AI should summarize only supplied data and should not infer facts that are not in the system.
- AI output should have deterministic fallback where possible.
- AI-generated recommendations should never be the only source of operational truth.

## 8. Data Model / Schema Needs

Phase 4 can start with existing tables. New tables should be deferred unless live usage proves they are needed.

### `generated_summaries`

- Necessary now: no.
- Purpose: generic storage for lead/reservation/post-sales summaries.
- Key fields: entity_type, entity_id, summary, risk_flags, recommended_actions, model, generated_by, created_at.
- Risks: duplicates existing `application_ai_reviews`, `customer_ai_summaries`, and `ai_daily_briefs`; may become too generic.
- Phase: later only if multiple new summary types need persistence.

### `smart_recommendations`

- Necessary now: no.
- Purpose: persistent recommendations across dashboards and entities.
- Key fields: source_type, source_id, title, details, severity, status, assigned_to, created_by, resolved_at.
- Risks: overlaps with `brief_action_items`, `follow_up_tasks`, and `post_sales_tasks`; could create duplicate task systems.
- Phase: later, or avoid by using existing task tables.

### `report_snapshots`

- Necessary now: no.
- Purpose: store historical report outputs for audit/comparison.
- Key fields: report_type, filters, metrics, generated_by, created_at.
- Risks: stale reports, storage growth, unclear retention policy.
- Phase: later.

### `daily_sales_briefs`

- Necessary now: no.
- Purpose: separate sales-only brief.
- Key fields: brief_date, sales_summary, alerts, recommended_actions, generated_by.
- Risks: duplicates `ai_daily_briefs`.
- Phase: avoid for MVP; extend `ai_daily_briefs` instead if schema can safely evolve later.

### `staff_action_items`

- Necessary now: probably no.
- Purpose: general action item tracking outside brief-generated items.
- Key fields: title, details, source_type, source_id, assigned_to, due_at, status, priority.
- Risks: duplicates `follow_up_tasks` and `post_sales_tasks`.
- Phase: later only if a unified action center is explicitly chosen.

MVP recommendation: avoid new tables. Use existing records, live queries, existing summaries, and existing task/checklist tables. If persistence is needed, extend existing AI summary tables in a later migration only after shape is confirmed.

## 12. Risks and Questions

Risks:

- AI cost can grow if summaries are generated frequently or across many records.
- AI hallucination could create false confidence if source data is unclear.
- Source data may be incomplete: assignments, lead source, activities, and task ownership depend on staff discipline.
- Permission boundaries must remain strict because AI summaries include private buyer/customer data.
- Over-alerting staff can reduce trust in the system.
- Duplicate summaries can confuse users if live rule-based panels and stored AI summaries disagree.
- Stale recommendations can remain visible if generated summaries are not refreshed.
- Reports can become too complex for small-team usage.
- Smart recommendations can duplicate follow-up tasks, post-sales tasks, and brief action items.
- Pending authenticated protected-route browser/mobile QA remains a release-quality gap.

Open questions:

- Should Daily Brief recommendations create actionable tasks, or remain separate carryover items?
- Should smart recommendations be generated live, stored, or converted into existing task records manually?
- Should Phase 4 extend `ai_daily_briefs` fields or keep sales/post-sales sections inside existing JSON arrays and text fields?
- Should staff workload reporting require every workflow record to have `assigned_to` before launch?
- Which sales stages and reservation statuses should count as stale or urgent for Belize/Caribbean operations?
- How often should AI summaries be regenerated, and who is allowed to refresh them?
- Should AI output include draft follow-up text for leads, or should that wait until messaging workflows are defined?
- Should reports be optimized for export, on-screen review, or both?
- What retention policy should apply if report snapshots or generated summaries are introduced later?
